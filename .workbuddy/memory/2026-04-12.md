## v2.1.10 修复详情

### 问题根源
虚拟行选择订单后，所有格子初始化为 `_dirtyCells[key] = 0`，但 `autoSaveWorkRecords` 只保存 `val > 0` 的记录，导致行从未被保存到数据库。重新加载后，行消失，回到"待选择状态"。

### 修复方案
1. 初始化改为 `_dirtyCells[key] = null`（表示已选择但未输入）
2. 立即保存一条 `quantity=0` 的记录标记这行已存在于数据库
3. 保存成功后重新加载 `workRecords`，再渲染表格
4. 确保虚拟行转换后，`renderSpreadsheet` 能正确显示实际行

## work-edit.js 彻底重写 (v2.2.3)

### 4 个 bug 根因分析

1. **改订单/型号后对数变**：`loadWorkRecords` 用 `order_id,model_id` 作行索引（comboKey），当 combo 变化时行 key 冲突/重新分组，导致数据被合并或丢失。  
   → 修复：每行分配永久唯一 `rowId`（递增），不受 combo 变化影响；combo 变化时清空 `emps`（不迁移），冲突时合并到已有行。

2. **新增行默认值不是空**：空行 `orderId:0/modelId:0` 与下拉不匹配，显示第一个选项的值而非空。  
   → 修复：`addWorkRow` 默认选第一个订单和型号（有效值），对数默认空对象 `emps:{}`，下拉正确显示。

3. **工资视角有时不计算**：直接用 `wageDetail.wages`（旧快照），而用户输入的 `_weRowMap` 已变化；且切换年月时 `wageDetail=null`。  
   → 修复：工资视角改为实时计算 `qty × price_map[modelId,subDeptId]`（基于当前 `_weRowMap`）；切换年月时如果已在工资视角则重新加载 `wage-detail`。

4. **切页后数据乱**：切换页面时 `loadWorkRecords` 重置状态，工资视角状态未保持。  
   → 修复：`loadWorkRecords` 先记录 `wasWageView`，切换年月时按需保持工资视角；工资视角下切页回来会重新加载。

### 核心设计变更

- `_workRowData` → `_weRowMap`，`rowIdx` → `rowId`（永久递增）
- `comboSeen` 仅在 `loadWorkRecords` 加载时用，重新渲染不走 combo key
- `onWorkSelectChange`：combo 变化时清空 `emps`，冲突时合并到目标行
- `calcCellWage`：工资视角实时计算，不依赖 `wageDetail.wages`
- 切换年月时保持工资视角状态（重载 `wage-detail`）

### v2.2.4 修复：删除后刷新数据回来

**根因**：`deleteWorkRow` 只删了 `_weRowMap`，`autoSaveWorkRecords` 只 upsert 格子记录（不删除），数据库中的 `work_records` 行数据从未被清理，刷新后 `loadWorkRecords` 又从 DB 读回来了。

**修复**：
1. 移除 `confirm()` 确认框
2. `deleteWorkRow` 改为 `async`，删除行时立即调 `DELETE /api/work-row?year=&month=&order_id=&model_id=`（批量删除该 combo 所有员工记录）
3. 后端 `/api/work-row` DELETE 接口已存在（`api_server.py` 第231行），直接复用

### v2.2.5 允许同 combo 多行

**设计变更**：每行新增 `lineDbId` 字段（DB `work_records.id`），区分同 `(orderId,modelId)` 的不同行。
- 加载时：每条 DB 记录独立成行（不再 `comboSeen` 分组合并）
- `lineDbId=null`：新增未保存的行
- `lineDbId!=null`：已保存的行，删除时调 `DELETE /api/work-row`
- 切换 combo 时：先删旧行（`DELETE /api/work-records` per emp），重建新行
- 保存时：后端 `save_work_record` 新增返回 `{"ok":true,"id":saved_id}`，前端据此回写 `lineDbId`

### v2.2.10 最终版：完全不合并，每行独立 line_id

**数据库**：`work_records` 新增 `line_id` 列，UNIQUE 约束改为
`(year, month, line_id, order_id, model_id, emp_id)`
→ 同 `(order_id, model_id)` 可以有多行，每行有独立 `line_id`

**前端**：
- `rowId`：前端唯一标识（DB行=r.id，新增行=_weRowCounter++）
- `lineId`：数据库行标识（DB行=r.line_id，新增行=_weMaxLineId++）
- 加载：每条 DB 记录独立一行（删掉 comboSeen 合并逻辑）
- 排序：`lineId DESC`（新增行 lineId 大 → 排前；DB 行 lineId 小 → 排后）
- 改下拉：删旧 lineId 行，分配新 lineId（lineId 递增，排到新增行之后）
- 删除：`DELETE /api/work-row?line_id=...`（精确到行）
- 保存：POST 带上 `line_id`，后端 UPSERT 到唯一约束

### v2.2.13 修复刷新后一行变多行（核心 key 体系重构）

**Bug 根因**：`_weRowMap` key 混用数字和字符串，导致查找失败，同一行多个员工的格子被拆成独立行。

**修复**：统一 map key 格式为字符串 `"orderId,modelId,lineId"`，并修正所有依赖链：
- `_weRowMap` key = `"${orderId},${modelId},${lineId}"`（DB行）或 `"new,0,${newLineId}"`（新增行）
- `renderSpreadsheet`：用 `Object.keys` 遍历，直接把 key 作为 `data-row` 属性传给事件处理器
- `onWorkCellChange`/`onWorkSelectChange`：直接用 `el.dataset.row`（字符串）作 map 查找
- `onWorkSelectChange`：combo 变化时用 `delete + create` 模式处理 key 重命名
- 排序：`"new"` 前缀优先，其余按 `lineId DESC`
